最近在折腾 OpenClaw，说实话还没完全搞明白怎么用，结果回头一看，人家已经出了 OpenClaw-RL 了。这速度也太快了，3月10日挂到 arxiv 上就直接冲上了 HuggingFace Daily Papers 第一名。


论文题目叫 OpenClaw-RL: Train Any Agent Simply by Talking，光看标题就很能抓人。核心思路说起来简单：你每次和 AI 的对话都会产生一个下一状态信号——你的回复、工具的输出、终端的执行结果。之前所有的 RL 系统都把这个信号扔掉了，OpenClaw-RL 说，不，我要把它用起来。


GitHub 仓库 Gen-Verse/OpenClaw-RL 目前已经有 1932 stars，代码在 3 月 10 日同步开源，支持 terminal、GUI、SWE、tool-call 四种 agent 场景的 RL 训练，跑在 Slime + Megatron 的基础上。


== 先说为什么要做这个 ==


传统的 RL 训练方式是离线的：先收集一批数据，然后拿去训练。这里面有两个信息被浪费掉了：一是评估信号（这次回答好不好），二是指令信号（这次回答哪里不对、应该怎么改）。


每一次 agent 和用户的交互，其实都包含这两类信息。用户重新问一遍，说明上次答得不够好。用户说你应该先检查一下文件再改，这就是明确的纠错指令。终端报错了，说明这步操作有问题。这些信息都在 next-state 里，但没有人去收集它。


== 算法：两种方法，一起用效果最好 ==


论文提出了两种从 next-state 信号里提取学习信号的方法，不是竞争关系，而是互补的。


Binary RL（二值强化学习）


这个方法把评估信号转成标量奖励。具体做法是用一个 PRM（过程奖励模型）来判断这次 action 好不好，打分为 +1、-1 或 0，跑 m 次然后多数投票，得到一个 final reward。


训练目标是 PPO-style 的 clipped surrogate，稍微改了一下上下界：ε=0.2，ε_high=0.28，β_KL=0.02。因为是实时对话场景，没有 GRPO 那种 group 结构可以做标准化，所以直接用 r_final 作为 advantage。


优点是覆盖面广，所有被打过分的 turn 都能进训练。缺点是信息太粗，一个标量没法告诉模型哪个 token 应该不同。


OPD：Hindsight-Guided On-Policy Distillation


这个方法针对的是指令信号，也是论文里最有意思的部分。


核心思路：如果用户说你应该先检查文件再改，这个信息如果在最开始就给模型（hint 放在 prompt 里），模型大概率就能生成更好的回答。那么，hint 增强后的 token 分布和没有 hint 的 token 分布之间的差距，就是每个 token 应该往哪个方向走的信号。


分四步走：

- Step 1. Hindsight hint 提取：用 judge 模型从 next-state 里提炼出 1-3 句精炼的纠错指令，放在 [HINT_START]...[HINT_END] 标签里。不直接用原始 next-state，因为用户的回复可能又包含新问题、又带着吐槽，很嘈杂。
- Step 2. hint 质量筛选：从多次 judge 调用里选最长的（信息量最多），长度不足 10 个字符的直接丢弃。没有有效 hint 的 sample 整个扔掉，宁缺毋滥。
- Step 3. 构建增强 teacher 上下文：把 hint 拼到最后一条 user message 后面，构成 s_enhanced，相当于如果用户一开始就给了这个提示，模型会生成什么。
- Step 4. Token 级别优势计算：A_t = log pi_teacher(a_t|s_enhanced) - log pi_theta(a_t|s_t)。A_t > 0 说明这个 token 在有 hint 的情况下模型更倾向于选，应该加强；A_t < 0 说明有了 hint 之后模型反而不选，应该压制。

这个方法的精妙之处：它不需要一个更强的 teacher 模型（不像 KD），也不需要人工标注偏好对（不像 DPO），teacher 就是自己加了 hint 之后的自己。


两者结合：Combined


最终的 advantage 是加权求和：A_t = w_binary * r_final + w_opd * (log pi_teacher - log pi_theta)，默认两个权重都是 1。Binary RL 提供广覆盖（所有被打分的 turn 都能训练），OPD 提供高精度（有明确纠错信息的 turn 给 token 级别的指导），两者不冲突，反而互补。


== 系统架构：四个组件完全异步 ==


这部分在论文里只写了几行，但代码里看起来工程量不小。整个系统有四个异步运行的组件：

- Policy Server：对外提供推理服务，接收来自 Environment 的请求
- Environment：模拟各种交互环境（terminal、GUI、web），不断产生 (state, action, next_state) 三元组
- PRM Judge：对每个 (s, a, s') 三元组打分，同时提取 hindsight hint
- Policy Trainer：消费打好标签的数据，异步更新模型权重，然后把新权重推给 Policy Server
