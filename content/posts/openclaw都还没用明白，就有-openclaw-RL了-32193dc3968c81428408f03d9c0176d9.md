最近在折腾 OpenClaw，说实话还没完全搞明白怎么用，结果回头一看，人家已经出了 OpenClaw-RL 了。这速度也太快了，3月10日挂到 arxiv 上就直接冲上了 HuggingFace Daily Papers 第一名。


论文题目叫 OpenClaw-RL: Train Any Agent Simply by Talking，光看标题就很能抓人。核心思路说起来简单：你每次和 AI 的对话都会产生一个下一状态信号——你的回复、工具的输出、终端的执行结果。之前所有的 RL 系统都把这个信号扔掉了，OpenClaw-RL 说，不，我要把它用起来。


GitHub 仓库 Gen-Verse/OpenClaw-RL 目前已经有 1932 stars，代码在 3 月 10 日同步开源，支持 terminal、GUI、SWE、tool-call 四种 agent 场景的 RL 训练，跑在 Slime + Megatron 的基础上。


== 先说为什么要做这个 ==


传统的 RL 训练方式是离线的：先收集一批数据，然后拿去训练。这里面有两个信息被浪费掉了：一是评估信号（这次回答好不好），二是指令信号（这次回答哪里不对、应该怎么改）。


每一次 agent 和用户的交互，其实都包含这两类信息。用户重新问一遍，说明上次答得不够好。用户说你应该先检查一下文件再改，这就是明确的纠错指令。终端报错了，说明这步操作有问题。这些信息都在 next-state 里，但没有人去收集它。


== 算法：两种方法，一起用效果最好 ==


论文提出了两种从 next-state 信号里提取学习信号的方法，不是竞争关系，而是互补的。


Binary RL（二值强化学习）


这个方法把评估信号转成标量奖励。具体做法是用一个 PRM（过程奖励模型）来判断这次 action 好不好，打分为 +1、-1 或 0，跑 m 次然后多数投票，得到一个 final reward。


训练目标是 PPO-style 的 clipped surrogate，稍微改了一下上下界：ε=0.2，ε_high=0.28，β_KL=0.02。因为是实时对话场景，没有 GRPO 那种 group 结构可以做标准化，所以直接用 r_final 作为 advantage。


优点是覆盖面广，所有被打过分的 turn 都能进训练。缺点是信息太粗，一个标量没法告诉模型哪个 token 应该不同。


OPD：Hindsight-Guided On-Policy Distillation


这个方法针对的是指令信号，也是论文里最有意思的部分。


核心思路：如果用户说你应该先检查文件再改，这个信息如果在最开始就给模型（hint 放在 prompt 里），模型大概率就能生成更好的回答。那么，hint 增强后的 token 分布和没有 hint 的 token 分布之间的差距，就是每个 token 应该往哪个方向走的信号。


分四步走：

- Step 1. Hindsight hint 提取：用 judge 模型从 next-state 里提炼出 1-3 句精炼的纠错指令，放在 [HINT_START]...[HINT_END] 标签里。不直接用原始 next-state，因为用户的回复可能又包含新问题、又带着吐槽，很嘈杂。
- Step 2. hint 质量筛选：从多次 judge 调用里选最长的（信息量最多），长度不足 10 个字符的直接丢弃。没有有效 hint 的 sample 整个扔掉，宁缺毋滥。
- Step 3. 构建增强 teacher 上下文：把 hint 拼到最后一条 user message 后面，构成 s_enhanced，相当于如果用户一开始就给了这个提示，模型会生成什么。
- Step 4. Token 级别优势计算：A_t = log pi_teacher(a_t|s_enhanced) - log pi_theta(a_t|s_t)。A_t > 0 说明这个 token 在有 hint 的情况下模型更倾向于选，应该加强；A_t < 0 说明有了 hint 之后模型反而不选，应该压制。

这个方法的精妙之处：它不需要一个更强的 teacher 模型（不像 KD），也不需要人工标注偏好对（不像 DPO），teacher 就是自己加了 hint 之后的自己。


两者结合：Combined


最终的 advantage 是加权求和：A_t = w_binary * r_final + w_opd * (log pi_teacher - log pi_theta)，默认两个权重都是 1。Binary RL 提供广覆盖（所有被打分的 turn 都能训练），OPD 提供高精度（有明确纠错信息的 turn 给 token 级别的指导），两者不冲突，反而互补。


== 系统架构：四个组件完全异步 ==


这部分在论文里只写了几行，但代码里看起来工程量不小。整个系统有四个异步运行的组件：

- Policy Server：对外提供推理服务，接收来自 Environment 的请求
- Environment：模拟各种交互环境（terminal、GUI、web），不断产生 (state, action, next_state) 三元组
- PRM Judge：对每个 (s, a, s') 三元组打分，同时提取 hindsight hint
- Policy Trainer：消费打好标签的数据，异步更新模型权重，然后把新权重推给 Policy Server

四个组件通过 Redis 或消息队列通信，互不阻塞。这样设计的好处是 GPU 利用率高——推理和训练可以同时跑，不用等数据收集完成。


== 代码实现：看完才知道工作量 ==


仓库目录结构很清晰，分了几个子项目：

- openclaw-rl/：核心 RL 训练框架，包含 Binary RL 和 OPD 的实现，基于 Slime
- terminal-rl/：终端场景的环境模拟，给模型一个沙箱 bash 环境跑命令
- gui-rl/：GUI 场景，截图 + 操作的 loop
- toolcall-rl/：工具调用场景，函数调用 + 返回结果的 loop
- Megatron-LM/：分布式训练基础设施

最关键的代码在 openclaw-rl/algorithms/ 下。Binary RL 的实现：收到 PRM 的打分后，直接把 r_final 当做 GAE 里的 return，然后用 clipped PPO loss 更新。注意这里没有 value function，是 policy gradient 的形式，和 REINFORCE 比较接近，但加了 KL 惩罚项。


OPD 的实现更有意思：先把 hint 拼到 prompt 里，过一遍 forward 得到 teacher 的 logits，再和当前 policy 的 logits 做差，得到 token-level advantage。具体来说：

- 只对 action token 计算 advantage，padding 和 system prompt 的 token 不算
- advantage 做了 per-sample 的 normalize，防止量级差异太大
- teacher forward 在 inference 模式下跑（不需要梯度），节省显存

== 实验结果 ==


论文在两类场景做了验证：个人代理（personal agent）和通用代理（general agent）。


个人代理实验：用 Qwen2.5-7B 作为 base model，在个人助手场景下用自己的历史对话记录做 RL 训练。三种方法对比：Binary RL 单独、OPD 单独、两者结合。结果是 Combined 效果最好，在 task completion rate 上比 base 提升了约 23%，在 preference win rate 上提升了约 18%。OPD 单独用比 Binary RL 单独用效果略好，两者结合提升比较明显，说明两种信号确实是互补的。


通用代理实验：用 Qwen2.5-72B 在 WebArena、OSWorld 等 benchmark 上测试。WebArena 上从 34.2% 提升到 41.7%，OSWorld 上从 12.8% 提升到 16.3%。这个提升幅度在 agent benchmark 上已经算不小了，因为这些 benchmark 本来就很难。


一个有意思的 ablation：把 PRM judge 换成 human label，效果差不多，说明 PRM 在这个任务上基本能代替人工打分。这对实际使用很重要——如果非得人工标注才能训练，那推广成本就太高了。


== 一些想法 ==


看完这篇论文，我觉得最有价值的地方不是 Binary RL 和 OPD 本身，而是它把一个问题说清楚了：agent 交互中的下一状态信号一直都在，只是没人去系统地利用它。


OpenClaw 原版解决的是让 AI 能用工具、能操作界面的问题，是能力的问题。OpenClaw-RL 解决的是让 AI 能从使用过程中持续变好的问题，是进化的问题。两者加在一起，思路上已经比较完整了。


但有几个问题我觉得还没有很好地解决：

- PRM 的质量是瓶颈。OPD 依赖 judge 提炼 hint 的质量，但 hint 提炼本身也是个难题。如果 judge 提炼出来的 hint 有偏差，OPD 的方向就错了。论文里用的是 GPT-4o 做 judge，实际部署时这个成本不低。
- 个人化 vs 通用性的取舍。personal agent 的训练数据是用户自己的历史，这能让模型更懂你，但也可能过拟合到你的习惯，泛化性变差。这个 tradeoff 论文里没有深入讨论。
- 隐私问题。用用户的历史对话做 RL 训练，数据怎么存、怎么用、能不能泄露，这些在论文里完全没提。但在实际产品里这是最大的阻碍之一。
- 训练稳定性。在线 RL 本来就容易不稳定，这个系统还是完全异步的，数据收集和训练的时序可能不对齐，会不会有 stale data 的问题，论文里没有详细讨论。

从更大的视角来看，OpenClaw-RL 代表的是一类思路：AI 系统不只是被部署，而是在部署中持续学习。这和传统的模型训练——收集数据、离线训练、发布——有根本性的区别。如果这条路走通了，AI 的迭代周期会从以月计缩短到以天甚至以小时计。


当然，这也意味着模型的行为会持续变化，可解释性和可控性的问题会更难。但这是后话了。


代码已经开源，有 GPU 资源的可以直接跑起来试试。就算跑不起来，读代码本身也值得——异步 RL 系统的工程实现比论文描述要复杂得多，细节都在代码里。
